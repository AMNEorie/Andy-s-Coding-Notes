# 1. 网络基础

## 1.1 网络架构

两种主要的网络架构：**C/S** 和 **B/S**。

### C/S：客户端 / 服务器（Client / Server）

需要下载客户端应用程序才能使用。

| 方面 | 描述 |
|------|------|
| **客户端** | 本地安装的终端程序（微信、抖音、Steam、英雄联盟） |
| **服务器** | 全天候待命，响应请求（如腾讯微信服务器） |
| **使用场景** | 游戏、银行应用（高性能和高安全需求） |

**优点：**
- 优秀的用户体验：丰富的图形/音频存储在本地
- 支持离线功能：单机游戏、文档编辑等
- 更好的安全性，数据可以存储在本地

**缺点：**
- 开发/维护成本更高（需要客户端和服务器）
- 用户必须下载更新
- 跨平台复杂性（iOS、Android、Windows）

### B/S：浏览器 / 服务器（Browser / Server）

无需安装，通过浏览器使用 URL 访问。

| 方面 | 描述 |
|------|------|
| **访问方式** | 浏览器 + URL（baidu.com、jd.com、bilibili.com） |
| **使用场景** | 娱乐、购物、网页游戏（注重便利性） |

**优点：**
- 无需开发客户端（只需要网页 + 服务器）
- 用户零安装，打开浏览器即可使用
- 更新方便：只需服务器端更新，用户刷新即可
- 跨平台：任何有浏览器的设备都能使用

**缺点：**
- 所有内容从服务器加载 → 依赖网络
- 大型应用性能较差（画质/音质较低）
- 与原生应用相比交互性受限

### 对比总结

| 标准 | C/S 架构 | B/S 架构 |
|------|----------|----------|
| **使用方式** | 需要下载 | 仅需浏览器 |
| **用户体验** | 快速、丰富、流畅 | 受浏览器限制，可能卡顿 |
| **跨平台** | 需要多个版本 | 任何浏览器都能运行 |
| **维护** | 需要更新客户端 | 仅服务器端更新 |
| **安全性** | 更安全（本地数据） | 依赖服务器，易受攻击 |
| **示例** | 英雄联盟、银行应用 | 淘宝、京东、微博 |

> **结论：** C/S 适合性能/安全要求高的场景（游戏）；B/S 适合易用性要求高的场景（网页应用）。很多应用两种架构都采用。
